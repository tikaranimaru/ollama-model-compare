# reference — 落とし穴と対処

実運用（gemma4比較、M5 Max/128GB/Ollama 0.30.3）で踏んだ問題と対処をまとめる。

## 1. MoEモデルはタグ名と速度が一致しない

`gemma4:26b` は「26B総パラメータ・活性≈4B」のMoE。`ollama show` の parameters は小さく出るが詐称ではない。
- ディスクサイズ ≈ 総パラメータ（全エキスパート保持）、生成速度 ≈ 活性パラメータ。
- なので 26B-MoE は 12B密モデルより**速い**ことがある。速度は「サイズ」でなく「活性パラメータ」で見積もる。

## 2. thinkingモデルは num_predict 不足で可視応答が空になる

gemma4等は推論(thinking)モデル。`num_predict` を思考トークンで使い切ると `response` が空・`done_reason=length` になる。
- 対処: `num_predict` を十分大きく（コード生成なら 12000〜16000）。
- 速度比較を公平にするなら `think:false` を付ける。
- generate.py は length 検知時に think 無効で自動再生成する。

## 3. ウォームアップの暴走（重要）

「モデルをロードするだけ」のウォームアップで `"hi"` を投げる際、`num_predict` が大きいと、
チャット最適化が弱いモデル（コード特化・base等）は自然停止せず上限まで生成し続け、
数十分ハングしたように見える。**ウォームアップは必ず `num_predict=1`**（generate.py は対応済み）。

## 4. 新しい量子化は古いOllamaで pull 不可（412）

`*-it-qat`（QAT）など新フォーマットは、古いOllamaだと
`pull model manifest: 412: ... requires a newer version of Ollama` で失敗する。
- 対処: Ollama をアップデート。ただし**アップデートはサーバー再起動を伴い実行中の生成を巻き込む**ので、
  生成が全部終わってから行う。`nvfp4 / mxfp8 / mlx` 系は比較的新しめでも取得できることが多い。

## 5. bf16 大型モデルは実用外＋"Stopping..."ハング

`*-coding-mtp-bf16`(64GB) は ~7 tok/s、1課題20分級。さらに:
- リクエストを途中で kill すると、`ollama ps` がそのモデルを `UNTIL: Stopping...` のまま固定表示する
  （ランナープロセスは消えているのにスケジューラに幽霊エントリが残る表示バグ／ハング）。
- 新規生成リクエストがこの停止処理待ちでブロックされることがある。
- **復旧**: 小さなリクエスト（`num_predict:5`）を投げて新ランナーを起動させると解消する。
  それでもダメなら `ollama stop <model>`、最終手段はOllamaサーバー再起動。
- 実務的には、超低速モデルは比較から外すか `--no-think-substr` で上限を絞る。

## 6. 単一GPUでは並行生成しても合計時間は縮まない

Apple Silicon等の単一GPUでは、複数モデルの同時生成は時間スライスになり総スループットは不変。
- 並行起動はモデルのロード/アンロードthrashを招くだけ。**逐次実行**が基本。
- 表示を早く埋めたいときだけ「速いモデルを先頭」に並べ替える（総時間は変わらないが体感が良い）。

## 7. プロンプト処理(pp) tok/s の見方

同一プロンプトの2回目以降はKVキャッシュ再利用で pp tok/s が数千〜数万に跳ねる。
- **pp は各モデル初回（冷値）のみ信頼**。bench.sh は1回計測なので冷値が出る。

## 8. 実効tok/s と ピークtok/s は乖離する

`bench.sh`（短文300トークン・think無効）はピーク値、`metrics.json`（長文コード＋think）は実効値。
長い出力ほどKVが伸びて per-token が落ちるため、実アプリ生成は数分の一になることがある。
**両方を併記**すると誤解が少ない。
